📋 Project Context: FastAPI Hybrid-Cloud CI/CD (GitHub Actions + Minikube)

Current Architecture: A high-concurrency Retail Store API (FastAPI, Python 3.11, PostgreSQL 15). The infrastructure runs locally on a Windows 11 machine using WSL2, Docker Desktop, and Minikube (allocated 6 CPUs, 8GB RAM).
Traffic Flow: External traffic hits an Nginx Ingress Controller (retail-store.local) -> routes to a ClusterIP Service -> load-balances across 3 API Pods (each running 2 Uvicorn workers).
The Orchestrator: We are bypassing GitHub's cloud runners to avoid network isolation. We are using a GitHub Actions Self-Hosted Runner installed directly inside the local WSL2 Ubuntu environment. This allows GitHub to securely trigger deployments on the local Minikube cluster.
The Goal: Generate a complete .github/workflows/deploy.yml pipeline that integrates Pytest unit tests, JMeter load testing (batches.jmx), and zero-downtime Kubernetes deployments.

⚙️ Pipeline Requirements (deploy.yml)
The workflow must trigger on pushes to the main branch, use runs-on: self-hosted, and execute the following sequential steps:

Checkout: Pull the latest repository code.

Unit Testing: Install requirements.txt and run pytest tests/. Abort if tests fail.

Performance Testing (JMeter): * Execute the JMeter test plan in CLI mode: jmeter -n -t batches.jmx -l results.csv.

Constraint: Run JMeter using an official Docker container (e.g., justb4/jmeter) mapped to the repository workspace so we don't have to install Java/JMeter natively on the WSL runner.

Execute the custom tests/check_performance.py script to evaluate results.csv. Abort the pipeline if the error rate > 0% or if throughput drops below acceptable thresholds.

Build & Register Image: Point the runner's Docker CLI to Minikube's internal daemon (eval $(minikube docker-env)) and build the retail-api:latest image so K8s can access it instantly.

Deploy: Execute kubectl rollout restart deployment retail-api-deployment.

Verify: Use kubectl rollout status to guarantee all 3 replicas spin up and pass their /health readiness probes without dropping traffic.

🚀 Required Deliverables from the LLM

The Workflow YAML: Generate the complete, production-grade .github/workflows/deploy.yml file matching the exact steps and constraints above.

The Evaluation Script: Provide the Python code for tests/check_performance.py using the standard csv module to parse the JMeter results, calculate the error rate, and enforce a sys.exit(1) hard-fail if performance degrades.

Runner Execution: Provide a brief reminder on how to ensure the WSL2 self-hosted runner has the necessary permissions to run Docker and Kubectl commands.

How the JMeter Integration Works
By asking the LLM to run JMeter via a Docker container in Step 3, you keep your WSL2 environment completely clean. You won't need to install Java Development Kits or configure JMeter system variables. The pipeline will pull a temporary JMeter container, map your batches.jmx file into it, fire the 10,000 requests against your Nginx Ingress, output the CSV, and then instantly destroy the JMeter container.

The Python script (check_performance.py) then acts as your automated Quality Assurance engineer, strictly enforcing the 0.00% error rate you achieved earlier before allowing the K8s deployment to proceed.
