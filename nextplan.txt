**🐘 Project Context: FastAPI Retail Store Kubernetes Migration**

**Current State:** A full-stack Retail Store inventory and purchasing API with a unified backend and Jinja2 frontend.
**Recent Optimizations:** The application was recently migrated from SQLite to PostgreSQL to resolve I/O bottlenecks. Extensive JMeter stress testing (10,000 concurrent requests) revealed that the optimal hardware configuration is 6 total Uvicorn workers. The SQLAlchemy connection pool is explicitly tuned (pool_size=20, max_overflow=30) to prevent exhausting PostgreSQL's max_connections (set to 350). The API is currently split into Public Customer routes and Protected Admin routes (secured via bcrypt and JWT authentication).
**The Goal:** Migrate the infrastructure from a single-node Docker Compose setup to a local Kubernetes (K8s) cluster. The primary objective is to implement a Horizontal Scaling architecture that supports Zero-Downtime deployments via K8s Rolling Updates.

**🛠️ The Tech Stack**

* **Backend:** FastAPI, Python 3.11
* **Server:** Uvicorn
* **Database:** PostgreSQL 15 (Official Alpine Docker Image)
* **Authentication:** JWT (python-jose), bcrypt (passlib)
* **Frontend:** Jinja2Templates (serving HTML/Vanilla JS directly from FastAPI)
* **Target Infrastructure:** Kubernetes (Minikube / Docker Desktop)

**📋 The Target Kubernetes Architecture**
The migration requires transitioning from a vertically scaled Docker setup (1 container, 6 workers) to a horizontally scaled Kubernetes setup. To respect local hardware CPU limits while maintaining the optimized 6-worker throughput, the architecture must be distributed:

1. **Database Layer (`postgres.yaml`):**
* Requires a `PersistentVolumeClaim` (2Gi).
* Requires a `Deployment` (1 replica) using `postgres:15-alpine`.
* Must override the Postgres initialization command to set `max_connections=350`.
* Requires an internal K8s `Service` named `db` mapping port 5432.


2. **API Layer (`api.yaml`):**
* Requires a `Deployment` targeting 3 replicas (Pods).
* *Worker Math:* Each Pod must run exactly 2 Uvicorn workers (3 Pods x 2 workers = 6 total system workers).
* Must implement a `RollingUpdate` strategy (`maxSurge: 1`, `maxUnavailable: 0`) to guarantee zero downtime.
* Must include `readinessProbe` and `livenessProbe` pointing to a `/health` endpoint on port 8000.
* Requires a `NodePort` Service mapping target port 8000 to node port 30000.
