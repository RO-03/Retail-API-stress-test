🐘 Project Context: FastAPI K8s Ingress Migration

Current State: A full-stack Retail Store inventory API running on a local Minikube cluster (allocated 6 CPUs and 8GB RAM on a Windows 11 host with WSL2). The system runs a PostgreSQL database (max_connections=350) and an API Deployment (3 Pods running 2 Uvicorn workers each).
The Bottleneck: Currently, JMeter load testing (10,000 concurrent requests) is being routed directly through a Kubernetes NodePort service. Throughput is capped at ~146 req/sec due to kube-proxy and iptables overhead.
The Goal: Migrate the external entry point from a NodePort to an Nginx Ingress Controller to achieve production-grade load balancing, bypass kube-proxy, and maximize throughput back to the 270+ req/sec baseline.

🛠️ The Tech Stack

Backend: FastAPI, Python 3.11, Uvicorn

Infrastructure: Minikube (Docker driver, WSL2 backend)

Target Router: Nginx Ingress Controller (Minikube Addon)

Target Hostname: retail-store.local

📋 The Migration Plan
The migration requires locking down the internal API Service and exposing it solely through the Nginx Ingress routing table.

Enable the Controller: Use Minikube's built-in addon system to deploy the Nginx Ingress Controller.

Secure the API Layer (k8s/api.yaml): Convert the existing retail-api-service from type NodePort to type ClusterIP. This removes direct external access, forcing all traffic to pass through the Nginx proxy.

Define the Routing Rules (k8s/ingress.yaml): Create a new Kubernetes Ingress manifest. It must listen for the host retail-store.local on the root path (/) and route traffic to the retail-api-service on port 8000.

Local DNS Mapping: Provide the instructions for updating the Windows 11 hosts file to map retail-store.local to the Minikube IP.
